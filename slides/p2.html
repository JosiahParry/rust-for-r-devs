R’s C API

  • Most languages know how to interact with C and C-types
  • R is literally written in C
  • Provides a C API

What is extendr?

Objective

Learn the different components of the extendr ecosystem.

extendr

extendr is a project that focuses on extending R with Rust. It is comprised of:

  • rextendr
  • extendr-api
  • extendr-ffi
  • extendr-engine

Note

There is no single “extendr” library. Rather, that is the name of the project.

extendr-ffi

  • Defines Rust types corresponding to R’s C API
  • Ensures Rust bindings are backwards compatible with C API changes
  • We will not touch this 😅

extendr-api

  • High level bindings built upon extendr-ffi
  • Feels natural from a Rust perspective
  • Handles R oddities e.g. garbage collection
  • We will only use extendr-api

extendr-engine

  • Calling R from Rust
  • Lets us start an R process from inside of Rust
  • Great for CLI tools

{rextendr}

  • {usethis}-like for developing Rust based packages
  • Primarily for:
    • scaffold R packages
    • document Rust functions
    • prepare packages for CRAN release

Adding the url crate

Objective

Add a dependency to our package and import items from it.

Declaring dependencies

  • R lets you use packages on your machine without declaring them
  • Rust will not
  • You must declare dependencies

Cargo.toml

  • Akin to the DESCRIPTION file
  • Dependencies set via the [dependencies] table
  • Add them via:
    • cargo add crate-name
    • or rextendr::use_crate("cratename")

Cargo.toml

[package]
name = "rurl"
publish = false
version = "0.1.0"
edition = "2021"
rust-version = "1.65"

[lib]
crate-type = [ "staticlib" ]
name = "rurl"

[dependencies]
extendr-api = "*"

Importing from dependencies

  • Items from deps are accessed via namespace, similar to R
  • Format crate_name::module::item
  • The * is a wildcard for importing everything
  • The prelude re-exports commonly used items
use extendr_api::prelude::*;
// import multiple items using `{}`
use crate_name::{StructName, function_name};

R’s Type System

Objective

Get a basic grip on R’s type system and develop an intuition for how Rust types map to R types.

Everything is an SEXP

  • To extendr, an SEXP is an Robj—it can be anything

Vector SEXP types

  • Integers are i32 in Rust
  • Doubles are f64 in Rust

Scalar types

  • i32 and f64 do not have NA
  • extendr-api provides:
    • Rfloat
    • Rint
    • Rstr
    • Rbool

Creating scalars

  • Created using ScalarType::from()
  • NAs created via ScalarType::na()
  • Check NA using x.is_na()
  • Access the inner type with .0
let my_float = Rfloat::from(3.14);
let my_na = Rfloat::na();
my_na.is_na() // returns true

Wrapper types

To work with vectors, we work with their “wrapper” types:

  • Doubles
  • Integers
  • Logicals
  • Strings

Always prefer these over Vec<f64> or Vec<i32>.

When in doubt, clone it out

Note

Cloning an extendr Robj (or other wrapper type) is cheap.

  • Use .clone() when something has been “moved”
  • Technically “expensive” for Vec<T> but still fast!

New types

Objective

Create your own struct from a “foreign” one.

The orphan rule

  • Imagine Shape is owned by another crate
#[derive(Debug, Clone)]
enum Shape {
    Triangle,
    Rectangle,
    Pentagon,
    Hexagon,
}
  • Types owned by other crates cannot have a new impl
  • That is called the “orphan rule”

The orphan rule

  • Shape is foreign, so we can’t impl on it directly

Doesn’t compile

impl Shape {
    fn my_custom_method(&self) { ... }
}

Using Newtypes

  • Wrap type in our own struct via struct NewType(OtherType)
  • Now the type is ours, so we can impl on it

Does compile

#[derive(Debug, Clone)]
#[extendr]
struct RShape(Shape);

impl RShape {
    fn my_custom_method(&self) { ... }
}

Adding methods

  • We “own” RShape
#[extendr]
impl RShape {
    fn new(x: &str) -> anyhow::Result<Self> {
        match x {
            "triangle" => Ok(Self(Shape::Triangle)),
            "rectangle" => Ok(Self(Shape::Rectangle)),
            "pentagon" => Ok(Self(Shape::Pentagon)),
            "hexagon" => Ok(Self(Shape::Hexagon)),
            _ => Err(anyhow::anyhow!("unknown shape: {x}")),
        }
    }
}

Exporting our impl to R

  • Register the impl in extendr_module!
extendr_module! {
    mod package;
    impl RShape;
}

External pointers

Objective

Use an #[extendr] impl to store a Rust struct inside an R object.

What we want

  • We want to create a struct in Rust and hold onto it in R
  • This is possible through an external pointer

What is an external pointer?

  • Points 👉🏼 to memory somewhere else
  • Lets you use other languages’ objects

A constructor

  • We need a helper to create our struct from R
  • Add a new() method on the struct
impl RShape {
    fn new(x: &str) -> anyhow::Result<Self> {
        match x {
            "triangle" => Ok(Self(Shape::Triangle)),
            "rectangle" => Ok(Self(Shape::Rectangle)),
            "pentagon" => Ok(Self(Shape::Pentagon)),
            "hexagon" => Ok(Self(Shape::Hexagon)),
            _ => Err(anyhow::anyhow!("unknown shape: {x}")),
        }
    }
}

new() isn’t special, just a conventional name.

Calling it from R

  • Include the impl in extendr_module! {}
extendr_module! {
    mod mypackage;
    impl RShape;
}

Now RShape$new("triangle") returns an R object that holds a Rust RShape.

Vectorization

Objective

Create a list of Url pointers from a character vector.

Working with many

  • R is vectorized
  • Remember, no scalars!!
  • Vectorization means looping in a low level language

How do we store multiple pointers?

  • An external pointer isn’t any type of vector
  • Lists are vectors!!!
  • Each element is a pointer

Strings wrapper type

  • Use the Strings wrapper type
  • Use .into_iter() to iterate through it
  • Each element is an Rstr, an NA aware &str
  • It can be used in most places that &str can be used

Mental model as Vec<String>

Collecting into a list

  • We can .collect() directly into a List
  • Every wrapper type can be turned into an Robj (everything is an SEXP!)
  • Our external pointer can be turned into an Robj
#[extendr]
fn parse_shapes(shapes: Strings) -> List {
    shapes
        .into_iter()
        .map(|vi| RShape::new(vi).unwrap())
        .collect()
}

A print helper

Objective

Write a function that reads a list of external pointers and turns it back into a character vector.

Ugly printing

  • R can’t see inside an external pointer
  • It only sees the memory address
parse_shapes(c("triangle", "hexagon"))
#> [[1]]
#> <pointer: 0x600000abc120>
#>
#> [[2]]
#> <pointer: 0x600000abc480>

Accessing ExternalPtr<T> 🐠💨

  • Stored in a struct called ExternalPtr
  • Has an associated type—the “turbo-fish”
  • Each type has a try_from() that fallibly converts
  • extendr provides ExternalPtr::try_from()
impl RShape {
    fn get_extptr(robj: Robj) -> extendr_api::Result<ExternalPtr<Self>> {
        ExternalPtr::<Self>::try_from(robj)
    }
}

Iterating a list

  • Each item is a tuple (val, val)
  • Fixed size, each element can be a different type
  • .0 is the name (&str), .1 is the value (Robj)
  • We usually ignore the name with _

Example

#[extendr]
fn format_shapes(x: List) -> Strings {
    x.into_iter()
        .map(|(_, robj)| match RShape::get_extptr(robj) {
            Ok(v) => Rstr::from(format!("{:?}", v.0)),
            Err(_) => Rstr::na(),
        })
        .collect()
}

Integrating with {vctrs}

Objective

Create a vctrs compatible vector for our urls.

S3 to the rescue

  • We build List objects from Rust
  • So we can lean on R’s S3 system
  • S3 dispatches on an object’s classes
  • Two steps:
    • assign a class to the list
    • define an R generic

The vctrs_vctr class

  • {vctrs} gives a vctrs_vctr class
  • Sensible default vector behavior for free
  • Assign c("rurls", "vctrs_vctr", "list")

Setting the class from Rust

  • Use .set_class() on a mutable Robj
  • Needs mut—we modify the underlying object
  • Fallible, so we return a Result
res.set_class(&["rurls", "vctrs_vctr", "list"])?;

The R method

  • A class alone gives basic printing
  • For format_rurls() we need a real R generic
#' @export
format.rurls <- function(x, ...) {
    format_rurls(x)
}

Getters

Objective

Write a family of functions that call a method on each object and collect the result.

A formulaic pattern

  • Iterate the list
  • Recover the pointer with get_extptr
  • Call the method
  • Collect into the right R type, NA on failure

Example

#[extendr]
fn shape_coords(x: List) -> Integers {
    x.into_iter()
        .map(|(_, vi)| match RShape::get_extptr(vi) {
            Ok(shape) => Rint::from(shape.n_coords() as i32),
            Err(_) => Rint::na(),
        })
        .collect::<Integers>()
}

Finding the methods

  • Docs: https://docs.rs/url/latest/url/struct.Url.html#implementations
  • LSP autocomplete—type rurl.0. and let rust-analyzer list them
  • Easy ones return a plain &str
  • Others return an Option—next chapter

Mapping into an Option

Objective

Transform the value inside an Option (or Result) without unwrapping it.

Some methods return Option

  • e.g. host_str() returns Option<&str>
  • The Option is Rust’s version of an NA
  • Plain getters won’t work on these

.map() works on Option

  • Just like mapping over an iterator
  • Closure runs only on Some (or Ok)
  • None stays None
let maybe_num: Option<i32> = Some(10);
let doubled = maybe_num.map(|n| n * 2); // Some(20)

Owned strings for R

  • host_str() gives a borrowed &str
  • R needs an owned String
  • Map into the option and call .to_string()
rurl.0.host_str().map(|v| v.to_string())

Tip

Rstr::from(Option<String>) turns None into NA.

🐮 CoWs and Modifying in place

Objective

Modify a value held in an external pointer without mutating the original.

Mutability is alien to R

  • R uses copy-on-write (CoW) 🐮
  • Modifying doesn’t change in place
  • R copies first, then modifies the copy
  • Url has methods that modify in place
  • So we clone before we modify

A helper to clone

  • Get the pointer
  • Borrow with as_ref(), then .clone()
impl RUrl {
    fn clone_from_robj(robj: Robj) -> anyhow::Result<Self> {
        match ExternalPtr::<RUrl>::try_from(robj) {
            Ok(v) => Ok(v.as_ref().clone()),
            Err(e) => Err(anyhow!("Expected an object of class `rurl`:\n{e}")),
        }
    }
}

Mutable values

  • Mark the clone mut
  • Call a method taking &mut self
  • A single String applied to every url for now
pub fn set_host(&mut self, host: Option<&str>) -> Result<(), ()>

Recycling & errors

Objective

Recycle a scalar argument over a vector and throw an error when lengths don’t line up.

R users expect recycling

  • A length-1 value applies to every element
  • url_set_scheme(urls, "https") reuses "https"

Recycling with .cycle()

  • .cycle() repeats an iterator forever
  • Zip a cycled iterator against the urls
  • The shorter one keeps repeating
url.into_iter().zip(scheme.into_iter().cycle())

Length checks

  • Recycling is valid for a scalar or equal length
  • Anything else is a mistake
  • Stop the whole call with throw_r_error()
if (scheme.len() != url.len()) & (scheme.len() != 1) {
    throw_r_error("`scheme` must be a scalar or the same length as `url`");
}
